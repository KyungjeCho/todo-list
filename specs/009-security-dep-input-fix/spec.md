## User Scenarios & Testing *(mandatory)*

### User Story 1 - 의존성 업데이트를 통한 공개 취약점 제거 (Priority: P1)

시스템 운영자로서, 서드파티 의존성의 공개된 Critical/High 취약점(Header Injection, SSRF, Prototype Pollution, ReDoS, XML Injection)이 모두 해소되어야 한다. 이를 통해 애플리케이션이 공개적으로 알려진 공격 벡터에 노출되지 않도록 한다.

**Why this priority**: Critical(CVSS 10.0) 및 High 심각도 취약점이 프로덕션 의존성에 존재한다. 이 취약점들은 공개적으로 알려져 있고 악용 가능하며, 시스템에 대한 가장 높은 즉각적 위험을 나타낸다.

**Independent Test**: 업그레이드 후 의존성 감사(audit)를 실행하여 대상 패키지에서 critical/high 취약점 0건을 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 프론트엔드 HTTP 클라이언트에 Header Injection 취약점(CVSS 10.0)이 존재할 때, **When** 패치된 버전으로 의존성을 업그레이드하면, **Then** 보안 감사에서 해당 취약점이 더 이상 나타나지 않는다.
2. **Given** 백엔드 프레임워크에 Injection, Prototype Pollution, ReDoS 취약점이 존재할 때, **When** 패치된 버전으로 프레임워크 패키지를 업그레이드하면, **Then** 관련 CVE가 모두 해소된다.
3. **Given** 개발 의존성에 Critical JavaScript Injection 및 Prototype Pollution 취약점이 포함되어 있을 때, **When** 테스트 도구를 업그레이드하면, **Then** 프론트엔드와 백엔드 양쪽에서 감사 경고가 제거된다.
4. **Given** 전이적(transitive) 빌드 의존성에 XML Injection 취약점이 있을 때, **When** 의존성 해석을 패치된 버전으로 오버라이드하면, **Then** 감사에서 해당 취약점이 더 이상 나타나지 않는다.
5. **Given** 모든 의존성 업그레이드가 적용된 상태에서, **When** 기존 테스트 스위트를 실행하면, **Then** 모든 테스트가 회귀 없이 통과한다.

### User Story 2 - 메모 콘텐츠 길이 제한 (Priority: P2)

사용자로서, 시스템이 메모 콘텐츠에 합리적인 길이 제한을 적용하여 과도한 입력으로 인한 서비스 품질 저하나 악용(서비스 거부, 저장소 고갈)을 방지해야 한다.

**Why this priority**: 입력 크기 제한이 없으면 단일 악의적이거나 실수로 발생한 요청이 무제한 데이터를 제출할 수 있어 성능 저하나 저장소 남용을 초래할 수 있다. 이는 반드시 해소해야 하는 표준적인 입력 검증 공백이다.

**Independent Test**: 다양한 크기의 메모 콘텐츠를 제출하여 5,000자 초과 콘텐츠가 적절한 에러로 거부되고, 제한 이내의 콘텐츠는 정상적으로 수락되는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 5,000자 이하의 콘텐츠로 메모를 생성할 때, **When** 요청을 제출하면, **Then** 메모가 정상적으로 생성된다.
2. **Given** 사용자가 5,000자를 초과하는 콘텐츠로 메모를 생성할 때, **When** 요청을 제출하면, **Then** 시스템이 명확한 검증 에러(400 status)로 요청을 거부한다.
3. **Given** 사용자가 5,000자 이하의 콘텐츠로 메모를 수정할 때, **When** 요청을 제출하면, **Then** 메모가 정상적으로 수정된다.
4. **Given** 사용자가 5,000자를 초과하는 콘텐츠로 메모를 수정할 때, **When** 요청을 제출하면, **Then** 시스템이 명확한 검증 에러(400 status)로 요청을 거부한다.

### User Story 3 - 프로덕션 HTTPS 강제 (Priority: P2)

시스템 운영자로서, 프로덕션 빌드에서 HTTP 백엔드 URL을 사용할 경우 앱이 시작을 거부하여, 사용자 데이터가 암호화되지 않은 연결로 전송되지 않아야 한다.

**Why this priority**: 프로덕션에서 HTTP를 통해 사용자 데이터를 전송하면 가로채기에 노출된다. 빌드 시점의 안전장치를 통해 안전하지 않은 설정으로의 실수 배포를 방지하여 모든 사용자를 기본적으로 보호한다.

**Independent Test**: 프로덕션 빌드 컨텍스트에서 HTTP URL을 설정하고 앱이 설명적인 에러와 함께 시작을 실패하는지 확인하고, 로컬 개발에서 HTTP URL이 영향 없이 동작하는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 애플리케이션이 HTTPS 백엔드 URL로 설정되었을 때, **When** 프로덕션 빌드가 시작되면, **Then** 애플리케이션이 정상적으로 초기화된다.
2. **Given** 애플리케이션이 HTTP 백엔드 URL로 설정되었을 때, **When** 프로덕션 빌드가 시작되면, **Then** 애플리케이션이 초기화를 거부하고 HTTPS가 필요하다는 명확한 에러 메시지를 표시한다.
3. **Given** 애플리케이션이 HTTP 백엔드 URL로 설정되었을 때, **When** 개발 빌드가 시작되면, **Then** 애플리케이션이 정상적으로 초기화된다(개발 모드에 영향 없음).

---

### Edge Cases

- 메모가 정확히 5,000자로 제출되면? (수락되어야 한다.)
- 메모에 5,000자 경계 부근에서 multi-byte Unicode 문자(한국어 등)가 포함되면? (제한은 바이트 수가 아닌 문자 수 기준으로 적용된다.)
- 프로덕션에서 백엔드 URL 환경변수가 설정되지 않으면? (기본 폴백은 localhost HTTP URL이므로 프로덕션에서 HTTPS 강제 에러가 발생해야 한다.)
- 의존성 업그레이드가 breaking API 변경을 도입하면? (업그레이드는 동일 주 버전 내 minor/patch 버전이므로 breaking 변경이 예상되지 않으며, 테스트 스위트가 호환성을 검증한다.)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 직접 및 전이적 의존성의 공개된 critical 심각도(CVSS 9.0+) 취약점을 반드시 해소해야 한다.
- **FR-002**: 시스템은 직접 및 전이적 의존성의 공개된 high 심각도(CVSS 7.0+) 취약점을 반드시 해소해야 한다.
- **FR-003**: 시스템은 메모 생성 시 콘텐츠에 최대 5,000자 길이 제한을 반드시 적용해야 한다.
- **FR-004**: 시스템은 메모 수정 시 콘텐츠에 최대 5,000자 길이 제한을 반드시 적용해야 한다.
- **FR-005**: 시스템은 메모 콘텐츠가 최대 길이를 초과할 때 명확한 검증 에러를 반드시 반환해야 한다.
- **FR-006**: 시스템은 백엔드 URL이 HTTP를 사용할 때 프로덕션 모드에서 시작을 반드시 거부해야 한다.
- **FR-007**: 시스템은 개발 모드에서 HTTP 백엔드 URL을 제한 없이 반드시 허용해야 한다.
- **FR-008**: 변경 적용 후 모든 기존 기능이 회귀 없이 반드시 정상 동작해야 한다.

### Key Entities

- **Memo**: 날짜와 연결된 사용자 생성 텍스트 콘텐츠. `content` 필드에 최대 5,000자 길이 제약이 추가된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 업그레이드 후 대상 패키지에 대해 의존성 감사 도구에서 critical 또는 high 심각도 취약점 0건 보고.
- **SC-002**: 5,000자를 초과하는 콘텐츠의 메모 생성/수정 요청 100%가 검증 에러로 거부됨.
- **SC-003**: 5,000자 이하 콘텐츠의 메모 생성/수정 요청 100%가 정상 성공함.
- **SC-004**: HTTP 백엔드 URL을 사용하는 프로덕션 빌드가 100% 설명적인 에러와 함께 시작 실패.
- **SC-005**: HTTP 백엔드 URL을 사용하는 개발 빌드가 100% 정상 시작 성공(회귀 없음).
- **SC-006**: 모든 변경 적용 후 기존 자동화 테스트 전체 통과.

## Assumptions

- 의존성 업그레이드는 기존 코드베이스와의 하위 호환성을 유지하는 minor/patch 버전으로 제한한다.
- 메모 콘텐츠 5,000자 제한은 현재 모든 사용자 워크플로우에 충분하다(실제로 이 길이를 초과하는 기존 메모가 없다고 가정).
- 개발 모드 감지 메커니즘이 개발 빌드와 프로덕션 빌드를 안정적으로 구분한다.
- 기존 테스트 스위트가 의존성 업그레이드로 인한 회귀를 감지하기에 충분한 커버리지를 제공한다.
- 보안 감사 해소 대상은 감사에서 식별된 특정 패키지와 CVE에 한정한다 — 포괄적인 보안 강화(예: rate limiting, CSP 헤더)는 범위 밖이다.

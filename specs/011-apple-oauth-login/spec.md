# Feature Specification: Apple OAuth 2.0 로그인 활성화

**Feature Branch**: `011-apple-oauth-login`
**Created**: 2026-04-20
**Status**: Draft
**Input**: User description: "기존 OAuth 2.0에 애플 프로바이더를 추가하고 애플 로그인 기능을 활성화 한다."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 신규 사용자가 Apple 계정으로 첫 로그인 (Priority: P1)

처음 앱을 사용하는 사용자가 로그인 화면에서 "Apple로 계속하기" 버튼을 눌러 자신의 Apple 계정으로 즉시 계정을 생성하고 홈 화면으로 진입한다.

**Why this priority**: Apple App Store 심사 정책상, 써드파티 소셜 로그인(Google/Naver/Kakao)을 제공하는 앱은 iOS에서 반드시 "Sign in with Apple"을 동등한 옵션으로 제공해야 한다. 이 기능이 없으면 iOS 앱 업데이트가 심사에서 반려되어 다른 기능 릴리스까지 막힌다. 또한 Apple 사용자는 이메일 릴레이 기능으로 개인정보를 보호하며 로그인하기를 선호하므로, 이 옵션 부재는 전환율 손실로 직결된다.

**Independent Test**: iOS 디바이스에서 앱을 새로 설치한 뒤 이 앱에 한 번도 로그인한 적 없는 Apple 계정으로 로그인을 시도하여 15초 이내에 홈 화면에 도달하고 프로필 조회가 성공하는지 확인한다. Android 디바이스에서도 동일 시나리오가 웹 기반 인증 플로우로 완료되는지 검증한다.

**Acceptance Scenarios**:

1. **Given** 로그인 화면이 노출된 iOS 사용자, **When** "Apple로 계속하기" 버튼을 탭하고 Apple ID 인증을 완료한다, **Then** 신규 사용자 계정이 생성되고 홈 화면으로 진입하며 로그인 세션이 유지된다.
2. **Given** 로그인 화면이 노출된 Android 사용자, **When** "Apple로 계속하기" 버튼을 탭하고 Apple ID 웹 인증을 완료한다, **Then** 신규 사용자 계정이 생성되고 홈 화면으로 진입한다.
3. **Given** 사용자가 Apple에서 "내 이메일 가리기"(private relay)를 선택한다, **When** 첫 로그인을 완료한다, **Then** 릴레이 이메일(`@privaterelay.appleid.com`)이 유효한 이메일로 저장되고 로그인이 정상 완료된다.
4. **Given** 사용자가 Apple 인증 화면에서 이름 공유를 허용한다, **When** 첫 로그인을 완료한다, **Then** 제공된 이름이 사용자 프로필 이름으로 저장된다.

---

### User Story 2 - 기존 Apple 사용자가 재로그인 (Priority: P1)

이미 Apple 계정으로 가입한 사용자가 앱을 재실행하거나 로그아웃 후 다시 로그인할 때, 동일한 계정과 데이터에 원활히 재접속한다.

**Why this priority**: 재로그인 실패는 기존 사용자 이탈로 직결되며, Apple 로그인은 2회차부터 이름을 공유하지 않으므로 첫 로그인에서 수집한 프로필을 정확히 재사용해야 한다. 로그인 기능이 "동작한다"라고 인정받으려면 최초 로그인뿐 아니라 반복 로그인도 안정적이어야 하므로 P1이다.

**Independent Test**: User Story 1 완료 직후 로그아웃을 수행한 뒤, 동일한 Apple 계정으로 재로그인하여 기존 할일 데이터와 이름이 그대로 유지되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** Apple 계정으로 가입을 완료한 사용자가 로그아웃된 상태, **When** "Apple로 계속하기"를 다시 선택해 인증을 완료한다, **Then** 신규 계정이 생성되지 않고 기존 계정으로 로그인되며 기존 할일/설정이 보존된다.
2. **Given** 기존 Apple 사용자가 재로그인한다, **When** Apple이 2회차 로그인부터 이름을 제공하지 않는다, **Then** 첫 로그인 시 저장된 이름이 그대로 유지되고 빈 문자열로 덮어쓰지 않는다.

---

### User Story 3 - Apple 로그인 실패 시 복구 (Priority: P2)

사용자가 Apple 인증을 취소하거나, 네트워크 오류 또는 서버 구성 오류가 발생했을 때 명확한 안내를 받고 다시 시도하거나 다른 로그인 수단을 선택할 수 있다.

**Why this priority**: 실패 처리가 없으면 사용자는 앱이 "멈췄다"고 판단하고 이탈한다. 기존 Google/Naver/Kakao 로그인의 오류 처리 수준과 동일한 UX 일관성이 필요하다. 정상 경로(P1) 구현 이후 추가되는 복원력 계층이므로 P2다.

**Independent Test**: Apple 인증 화면에서 사용자가 "취소"를 선택했을 때, 그리고 서버가 의도적으로 오류 응답을 반환하도록 구성했을 때 각각 사용자에게 오류 메시지가 표시되고 로그인 화면이 재입력 가능한 상태로 복귀하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** Apple 인증 화면에 진입한 사용자, **When** 사용자가 인증을 취소한다, **Then** 로그인 화면으로 복귀하고 다른 프로바이더 버튼이 다시 활성화된다.
2. **Given** 네트워크가 끊긴 상태, **When** 사용자가 Apple 로그인을 시도한다, **Then** 다국어 오류 메시지가 표시되고 재시도가 가능하다.
3. **Given** 서버가 Apple 토큰 교환에 실패한다, **When** 콜백이 처리된다, **Then** 일반 사용자에게는 친화적 오류 메시지를, 운영 로그에는 원인 식별이 가능한 상세 로그를 남긴다.

---

### Edge Cases

- 사용자가 Apple 설정에서 앱에 대한 "Apple ID로 로그인" 권한을 취소(revoke)한 경우 다음 로그인 시 동작이 정의되어야 한다.
- Apple 릴레이 이메일을 사용하던 사용자가 Apple 설정에서 릴레이를 비활성화한 경우, 저장된 이메일의 업데이트 정책이 필요하다.
- 동일한 실제 이메일을 가진 사용자가 Google/Naver/Kakao 계정으로 이미 가입한 상태에서 Apple로 재가입을 시도할 때의 계정 처리 정책이 필요하다.
- Apple 인증 세션이 외부 브라우저 탭에서 완료되거나 타임아웃된 경우 앱이 무한 로딩에 빠지지 않아야 한다.
- Apple이 제공하는 고유 식별자(Subject Identifier)만으로 동일 사용자 인식이 가능해야 하며, 이메일 값이 비어 있거나 변경되더라도 같은 사용자로 식별되어야 한다.
- 사용자가 Apple 인증 완료 직후 앱을 백그라운드로 보내거나 강제 종료한 경우, 다음 실행 시 미완 상태를 감지하고 재로그인 유도 또는 복구가 가능해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 로그인 화면에서 기존 Google/Naver/Kakao와 동등한 위계로 "Apple로 계속하기" 옵션을 제공해야 한다.
- **FR-002**: 시스템은 사용자가 Apple 옵션을 선택했을 때 Apple의 공식 인증 흐름으로 진입시켜야 한다.
- **FR-003**: 시스템은 Apple로부터 받은 인증 자격 증명을 검증하여 사용자의 고유 식별자(Apple Subject Identifier)와 이메일(있을 경우)을 안전하게 수신해야 한다.
- **FR-004**: 시스템은 첫 로그인에서 Apple이 제공한 이름과 이메일(릴레이 포함)을 사용자 프로필로 저장해야 한다.
- **FR-005**: 시스템은 동일 Apple 계정으로 재로그인할 때 기존 사용자로 식별하여 새 계정을 생성하지 않아야 한다.
- **FR-006**: 시스템은 Apple이 2회차 이후 로그인에서 이름을 제공하지 않더라도 기존 사용자의 이름을 빈 값으로 덮어쓰지 않아야 한다.
- **FR-007**: 시스템은 Apple 릴레이 이메일(`@privaterelay.appleid.com`)을 유효한 이메일로 수락하고 저장해야 한다.
- **FR-008**: 시스템은 Apple 인증 실패(사용자 취소, 네트워크 오류, 서버 오류)를 감지하여 사용자에게 다국어(한국어/영어 등 기존 지원 언어) 친화적 메시지로 안내하고 재시도를 허용해야 한다.
- **FR-009**: 시스템은 Apple 로그인으로 생성된 계정에 대해서도 기존 OAuth 프로바이더와 동일한 로그아웃, 토큰 갱신, 세션 관리 동작을 일관되게 지원해야 한다.
- **FR-010**: 시스템은 Apple 인증 요청과 콜백 처리 과정에서 CSRF(위조 요청) 및 재생(replay) 공격을 기존 프로바이더와 동일한 수준으로 방어해야 한다.
- **FR-011**: 시스템은 Apple OAuth 설정 자격 증명(클라이언트 식별자, 서명 키 등)을 환경 구성 값으로 분리 관리하고 소스 코드 및 리포지토리에 노출하지 않아야 한다.
- **FR-012**: 시스템은 iOS와 Android 모두에서 기존 Google/Naver/Kakao와 동일한 웹 기반 인증 세션 방식으로 Apple 로그인을 처리해야 한다. 플랫폼별 네이티브 Apple 로그인 시트는 본 범위에서 사용하지 않으며, 기존 OAuth 리다이렉트 플로우를 그대로 재사용한다.
- **FR-013**: 시스템은 Apple 로그인 사용자를 이메일 값과 무관하게 항상 독립된 계정으로 생성·식별해야 한다. 동일 이메일이 기존 Google/Naver/Kakao 계정에 존재하더라도 자동 연결하지 않으며, 계정 식별은 `(provider, providerUserId)` 조합만을 기준으로 수행한다. 사용자 주도의 계정 연결 기능은 본 범위에 포함되지 않는다.
- **FR-014**: 시스템은 사용자가 Apple 설정에서 앱 권한을 철회(revoke)한 경우에도 클라이언트 측에서 무한 로딩이나 크래시 없이 로그인 화면으로 정상 복귀시켜야 한다.
- **FR-015**: 시스템은 Apple 로그인 완료율(성공/실패), 실패 원인 분류(취소, 타임아웃, 서버 오류), 신규/재가입 구분을 관찰 가능하도록 계측해야 한다.

### Key Entities *(include if feature involves data)*

- **OAuth 연결(OAuth Linkage)**: 한 사용자가 어떤 프로바이더의 어떤 고유 식별자로 로그인했는지를 기록하는 연결. Apple 추가로 `provider = "apple"`, `providerUserId = Apple Subject Identifier`가 신규 저장되며, 이메일은 릴레이 주소일 수 있다. (provider, providerUserId) 조합은 전역적으로 고유하다.
- **사용자 프로필(User Profile)**: 이메일, 이름 등 사용자 식별·표시 정보. Apple의 특수성으로 이름은 첫 로그인에서만 수집 가능하며, 이메일은 릴레이 주소일 수 있다.
- **인증 세션(Authentication Session)**: 로그인 성공 후 발급되는 액세스/리프레시 토큰 쌍과 디바이스/FCM 정보. Apple 로그인도 기존과 동일한 세션 모델을 공유한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: iOS 사용자가 Apple 계정으로 최초 로그인을 시작하여 홈 화면 진입까지 95% 이상의 시도에서 15초 이내에 완료한다(정상 네트워크 기준).
- **SC-002**: 신규 Apple 사용자의 로그인 성공률이 기존 Google 프로바이더 성공률 대비 ±5% 포인트 이내로 유지된다.
- **SC-003**: 기존 Apple 사용자가 재로그인할 때 새 계정이 잘못 생성되는 사고가 0건이며, 이름이 빈 값으로 덮어써지는 회귀가 0건이다.
- **SC-004**: Apple App Store의 iOS 앱 심사에서 "Sign in with Apple 부재"를 사유로 한 심사 반려가 기능 배포 이후 0건 발생한다.
- **SC-005**: Apple 로그인 실패 시 사용자에게 다국어 오류 메시지가 100% 노출되며, 지원 채널로 유입되는 Apple 로그인 관련 문의가 월 10건 미만으로 유지된다.
- **SC-006**: Apple 로그인 경로의 인증 자격 증명(서명 키 등)이 리포지토리, 빌드 산출물, 로그에 평문으로 노출되지 않는다(정기 스캔 0건).

## Assumptions

- 기존 OAuth 2.0 인증 인프라(상태 서명, 콜백 라우팅, 토큰 발급, 세션/디바이스 관리, i18n 오류 처리)를 그대로 재사용하며, Apple만을 위한 별도 인프라를 구축하지 않는다.
- Apple Developer 계정과 필요한 식별자(App ID, Services ID, Sign in with Apple 활성화), 인증용 서명 키가 릴리스 전에 발급·구성된다고 가정한다. 자격 증명의 발급 절차 자체는 본 기능의 스코프가 아니다.
- 데이터베이스 스키마는 변경 없이, 기존 OAuth 연결 테이블의 `provider` 값으로 `"apple"`을 저장한다. 신규 컬럼/테이블은 도입하지 않는다.
- 지원 대상 플랫폼은 iOS와 Android 모두이며, 웹 전용 사용 사례는 본 기능 범위에서 제외된다.
- Apple이 제공하지 않는 프로필 항목(전화번호, 프로필 사진 등)은 요구하지 않으며, 사용자는 앱 내 설정에서 추후 보완할 수 있다.
- "내 이메일 가리기" 릴레이 이메일은 본 앱의 이메일 기반 기능에서 정상적인 주소로 취급되며, 별도의 실제 이메일 확인 절차는 도입하지 않는다.
- 데이터 보관 정책, 세션 타임아웃, 토큰 수명, 재시도 정책은 기존 프로바이더와 동일하게 따른다.
- 기존 저장소에 Apple 관련 코드 스캐폴딩이 일부 존재하더라도, 본 기능은 실제 동작하는 수준의 end-to-end 활성화를 목표로 한다.
- iOS 구현은 기존 프로바이더와 동일한 웹 기반 인증 세션 방식을 사용한다. Apple 권장 네이티브 시트는 사용하지 않으며, 이로 인한 Apple 심사 관련 리스크는 사후 모니터링하고 필요 시 후속 기능으로 분리한다.
- Apple 로그인 사용자는 기존 프로바이더 계정과 이메일이 같더라도 독립 계정으로 분리한다. 동일 인물이 여러 계정을 보유할 수 있으며, 본 기능에서는 계정 연결/통합 UI를 제공하지 않는다.

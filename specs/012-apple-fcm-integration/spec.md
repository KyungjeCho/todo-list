# Feature Specification: 애플 기기 FCM 연동

**Feature Branch**: `012-apple-fcm-integration`
**Created**: 2026-04-21
**Status**: Draft
**Input**: User description: "애플 기기 FCM 연동하기"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - iOS 사용자가 Todo 알림을 실제로 수신 (Priority: P1)

iPhone/iPad로 todo-list 앱을 사용하는 사용자가 로그인한 뒤 일정(Plan) 및 회고(Review) 알림을 iOS 알림 센터에서 수신한다. 화면을 끈 상태(잠금 화면)와 백그라운드, 앱이 포그라운드인 상태 모두에서 알림이 도착해야 하며, 알림을 탭하면 앱이 열리고 해당 Todo/회고 화면으로 이동한다.

**Why this priority**: iOS 사용자는 현재 FCM 토큰이 등록되어도 실제로 푸시가 전달되지 않는 상태다(서버는 보내지만 Apple APNs 연동/권한/프로비저닝이 완성되지 않아 장치에 도달하지 않음). 이 시나리오가 성립해야 iOS 사용자에게 알림 기능이 "켜졌다"고 말할 수 있다. 다른 시나리오는 이 기반 위에서만 의미가 있다.

**Independent Test**: 실제 iOS 기기(또는 TestFlight 빌드)로 로그인한 뒤 백엔드에서 테스트 알림을 트리거하여, 앱이 포그라운드/백그라운드/종료 상태 각각에서 알림을 수신함을 확인할 수 있다. Android 경로에 영향 없이 iOS만 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** iOS 사용자가 앱에 로그인하고 알림 권한을 허용한 상태, **When** 백엔드가 해당 사용자에게 Plan 유형 알림을 발송, **Then** 사용자의 iOS 기기 알림 센터에 제목/본문이 기대한 내용대로 10초 이내에 표시된다.
2. **Given** 앱이 백그라운드 상태, **When** Review 유형 알림이 도착, **Then** 잠금 화면/알림 센터에 알림이 표시되고 탭 시 앱이 회고 화면으로 진입한다.
3. **Given** 앱이 포그라운드 상태, **When** 알림이 도착, **Then** 앱은 현재 UI를 유지하면서도 알림 수신 이벤트를 처리(인앱 표시 또는 무시 규칙)하고 충돌하지 않는다.

---

### User Story 2 - 권한 요청 및 상태 관리 (Priority: P2)

iOS 사용자가 앱을 처음 로그인할 때 iOS 시스템 푸시 권한 다이얼로그가 1회 노출된다. 사용자가 "허용"을 누르면 이후 알림을 받고, "허용 안 함"을 누르면 앱은 충돌 없이 계속 동작하되 알림을 받지 않는다. 사용자가 iOS 설정에서 권한을 변경하면 앱 재실행 시 그 상태가 반영된다.

**Why this priority**: iOS 앱 심사 및 사용자 경험 관점에서 권한 다이얼로그의 맥락·타이밍은 중요하다. 권한 거부 경로가 안전하게 처리되지 않으면 토큰 미등록 상태에서 불필요한 백엔드 호출이나 에러 로그가 발생하고, 재인증 시 이중 프롬프트가 뜰 수 있다. P1보다 뒤지는 이유는 P1 기본 경로가 동작하지 않으면 권한 관리 자체가 의미 없기 때문.

**Independent Test**: 새 iOS 기기(또는 시뮬레이터 권한 리셋) 상태에서 앱을 실행하여 권한 다이얼로그가 1회만 표시되는지, 거부 시 앱이 정상 동작하는지 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** iOS 기기에서 앱을 처음 로그인, **When** 로그인 완료, **Then** iOS 시스템 알림 권한 다이얼로그가 1회 노출된다.
2. **Given** 사용자가 권한을 거부, **When** 앱을 계속 사용, **Then** 앱은 크래시 없이 동작하고 FCM 토큰 등록 호출은 발생하지 않는다.
3. **Given** 권한이 거부된 상태에서 사용자가 iOS 설정에서 권한을 허용으로 변경, **When** 앱을 재실행, **Then** 앱은 알림 권한이 허용된 상태를 감지하고 FCM 토큰을 서버에 등록한다.

### User Story 3 - 다기기·토큰 갱신 대응 (Priority: P3)

한 사용자가 iPhone과 iPad 두 기기 모두에서 로그인하면 두 기기 모두에 알림이 도착한다. 앱을 삭제했다가 재설치하거나 OS가 토큰을 갱신할 경우, 앱은 새 토큰을 서버에 등록하여 다음 알림부터 새 토큰 기준으로 전달된다. 사용자가 한 기기에서 로그아웃하면 그 기기로는 더 이상 알림이 전달되지 않는다.

**Why this priority**: 멀티 디바이스/토큰 갱신은 iOS에서 실제로 자주 발생하지만, 대부분 사용자가 1차 기기에서만 주로 쓰므로 P1/P2 완성 후 보강해도 된다.

**Independent Test**: 동일 계정으로 두 iOS 기기에 로그인하여 두 기기에 모두 알림이 수신되는지, 앱 재설치 후 기존 토큰으로의 전송이 실패하지 않고 새 토큰으로 전달되는지 확인 가능하다.

**Acceptance Scenarios**:

1. **Given** 한 사용자가 iPhone과 iPad 모두 로그인, **When** 백엔드가 알림 발송, **Then** 두 기기 모두에 동일 알림이 전달된다.
2. **Given** 앱을 삭제 후 재설치, **When** 다시 로그인, **Then** 새 FCM 토큰이 서버에 등록되고 기존 만료 토큰은 다음 발송 실패 시 정리된다.
3. **Given** 사용자가 iPad에서 로그아웃, **When** 백엔드가 알림 발송, **Then** iPad에는 알림이 전달되지 않고 iPhone에만 전달된다.

---

### Edge Cases

- 사용자가 iOS 시스템 "방해 금지(Focus / Do Not Disturb)"를 켠 상태에서 알림이 도착 → iOS 기본 동작에 위임(앱이 별도 처리하지 않음, 단 발송 자체는 성공 처리).
- 사용자가 네트워크 없음/기내 모드에서 알림이 발송됨 → APNs가 일정 시간 보관 후 재전송, 앱은 별도 조치 불필요하되 발송 로그는 성공 처리.
- Apple APNs 토큰이 만료(Unregistered)된 경우 → 서버는 해당 FCM 토큰 발송 시 오류를 받고 DB에서 해당 토큰 레코드를 정리한다.
- 권한 팝업에서 사용자가 "허용 안 함"을 선택한 뒤, 서버가 이미 구 토큰을 가지고 있는 경우 → 구 토큰은 다음 발송 실패 시 정리된다.
- 시뮬레이터에서 실행 (Apple Silicon 포함) → 푸시 미지원 상태가 감지되면 크래시 없이 건너뛰고 로그만 남긴다.
- 한 계정이 iPhone과 iPad 양쪽에 동일 FCM 토큰을 반환하지 않음(토큰은 기기별 고유) → 서버는 user_id+fcm_token 조합이 다수 존재함을 정상 상태로 취급한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 iOS 기기에서 로그인한 사용자에게 iOS 시스템 알림 권한을 요청해야 한다(1회).
- **FR-002**: 시스템은 사용자가 알림 권한을 허용한 경우에만 FCM 토큰을 획득하고 백엔드에 `deviceType='IOS'`로 등록해야 한다.
- **FR-003**: 시스템은 iOS 기기에서 발생하는 FCM 토큰 변경(OS 갱신, 앱 재설치 등)을 감지하고 변경된 토큰을 자동으로 재등록해야 한다.
- **FR-004**: 시스템은 한 사용자가 보유한 복수의 iOS 기기(예: iPhone + iPad)를 모두 구분해서 저장하고, 알림 발송 시 모든 활성 기기로 전달해야 한다.
- **FR-005**: 백엔드는 Plan 및 Review 알림 발송 시 iOS 기기로 발송한 결과가 APNs "Unregistered"/"BadDeviceToken" 등 만료 응답일 때 해당 토큰 레코드를 삭제(정리)해야 한다.
- **FR-006**: 포그라운드 상태에서 알림이 도착한 경우 시스템은 앱을 크래시시키지 않고 이벤트를 수신해야 한다. (인앱 표시 UI 자체는 이번 스코프에 포함되지 않는다.)
- **FR-007**: 알림 탭 시 시스템은 앱을 포그라운드로 전환하고 알림 유형(Plan/Review)에 해당하는 화면으로 이동해야 한다.
- **FR-008**: 시스템은 사용자가 iOS 설정에서 권한을 나중에 허용으로 변경한 경우, 앱 재실행 시 토큰을 획득·등록해야 한다.
- **FR-009**: 시스템은 사용자가 로그아웃할 때 해당 기기의 FCM 토큰을 서버에서 해제(해당 기기로 더 이상 전송되지 않도록)해야 한다.
- **FR-010**: 시스템은 권한 거부·시뮬레이터 등 토큰 획득 불가 상황에서 충돌이나 반복 재시도 없이 조용히 스킵해야 한다.
- **FR-011**: 시스템은 iOS 발송 경로 도입이 기존 Android 발송의 성공률/지연에 영향이 없음을 보장해야 한다(회귀 방지).
- **FR-012**: 시스템은 FCM 토큰, APNs 인증 키, Apple 개발자 팀 ID 등 민감 값을 로그에 평문으로 남기지 않아야 한다(토큰은 프리픽스만 기록 허용).

### Key Entities *(include if feature involves data)*

- **UserDevice(기존)**: 사용자별 등록된 푸시 수신 기기. `userId`, `fcmToken`(고유), `deviceType`('IOS' | 'ANDROID'), `deviceName`(선택). 본 기능으로 엔티티 스키마 변경은 필요하지 않으며, `deviceType='IOS'` 레코드가 실효적으로 생성·유지되는 흐름을 완성하는 것이 본 범위다.
- **NotificationLog(기존)**: 발송 이력. iOS 발송 결과(성공/만료/실패)가 Android와 동일한 로그 포맷으로 남는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 내부 테스트에서 iOS 기기로 발송한 푸시 알림의 95% 이상이 10초 이내에 기기에 도착한다(포그라운드/백그라운드/잠금 상태 평균).
- **SC-002**: 알림 권한을 허용한 iOS 사용자의 99% 이상이 FCM 토큰 등록에 성공한다(첫 로그인 ~ 30초 이내).
- **SC-003**: 기능 출시 이후 1주일 동안 iOS 기기의 푸시 발송 실패율(만료 토큰 정리 후 재계산)이 2% 이하를 유지한다.
- **SC-004**: 기능 출시로 인한 Android 푸시 발송 성공률 회귀가 없다(배포 전/후 성공률 차이 0.5%p 이하).
- **SC-005**: 알림 권한 거부 또는 시뮬레이터 실행 시 알림 초기화 코드 경로에서의 크래시/에러 로그가 0건이다(iOS 실기기 1대 + iOS 시뮬레이터 1대를 최신 Xcode stable 로 QA 4시간 세션 기준).
- **SC-006**: 사용자가 로그아웃하면 해당 기기로의 이후 푸시 발송이 100% 차단된다.

## Assumptions

- 지원 OS: iOS 15 이상. Expo ~55 최소 요구치이며 APNs Auth Key(`.p8`) 방식은 iOS 10 이후 모든 버전과 호환되므로 본 전제는 실질 제약을 추가하지 않는다.
- Firebase 프로젝트에 iOS 앱이 등록되어 있고, APNs 인증 키(`.p8`, Key ID, Team ID 세트)가 업로드 가능하다. 업로드 자체는 운영/인프라 범위에서 수행된다.
- Apple Developer 계정과 앱 번들 ID(iOS)가 이미 존재하며 Push Notifications capability와 프로비저닝 프로파일 갱신 권한이 있다.
- 기존 `notification` 모듈(`UserDevice`, `FcmService`, `register-device`, `send-notification`)을 재사용하며 스키마 변경은 하지 않는다.
- Android 경로의 FCM 연동은 이미 정상 동작 중이며 본 기능의 참조/회귀 기준으로 삼는다.
- 본 기능의 범위는 Plan/Review 등 기존 알림 유형의 iOS 전달을 완성하는 것이다. 새로운 알림 유형(배지, 마케팅, 위치 기반 등) 추가 및 Rich Notification(이미지/액션 버튼)은 범위에 포함되지 않는다.
- 사일런트/백그라운드 데이터 전용 푸시(content-available)는 범위에 포함되지 않으며, 표준 사용자 가시 알림만 다룬다.
- 인앱 포그라운드 알림 UI(토스트/배너)의 시각 디자인은 별도 기능으로 분리한다. 본 기능은 이벤트 수신/충돌 방지까지를 보장한다.

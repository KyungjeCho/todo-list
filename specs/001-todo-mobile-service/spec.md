## User Scenarios & Testing *(mandatory)*

### User Story 1 - OAuth 인증 및 온보딩 (Priority: P1)

사용자가 앱을 처음 설치하고 OAuth 소셜 로그인(Google, Naver, Kakao, Apple)을 통해 계정을 생성한다. 신규 사용자는 온보딩 화면에서 계획 시간과 회고 시간을 설정한 후 메인 화면에 진입한다. 기존 사용자는 로그인 후 바로 메인 화면으로 이동한다.

**Why this priority**: 인증은 모든 기능의 전제 조건이며, 온보딩은 루틴 기반 앱의 핵심 초기 설정이다. 이것 없이는 다른 어떤 기능도 사용할 수 없다.

**Independent Test**: OAuth 로그인 → 신규/기존 사용자 분기 → 온보딩 완료까지 테스트 가능하며, 이것만으로 사용자 계정 생성이라는 가치를 전달한다.

**Acceptance Scenarios**:

1. **Given** 앱이 설치된 상태에서, **When** 사용자가 소셜 로그인 버튼(Google/Naver/Kakao/Apple)을 탭하면, **Then** 해당 provider의 인증 페이지로 이동하고, 인증 완료 후 JWT 토큰이 발급된다.
2. **Given** 신규 사용자가 로그인에 성공하면, **When** 앱으로 돌아오면, **Then** 온보딩 화면이 표시되고 계획 시간(기본 08:00)과 회고 시간(기본 22:00)이 미리 채워져 있다.
3. **Given** 온보딩 화면에서 시간을 설정한 상태에서, **When** 설정 완료 버튼을 탭하면, **Then** 설정이 저장되고 메인 화면으로 진입한다.
4. **Given** 기존 사용자가 로그인에 성공하면, **When** 앱으로 돌아오면, **Then** 온보딩 없이 바로 메인 화면으로 진입한다.
5. **Given** 사용자가 로그인된 상태에서, **When** 토큰이 만료되면, **Then** Refresh Token으로 자동 갱신되어 끊김 없이 서비스를 이용할 수 있다.

### User Story 2 - 할 일 CRUD 및 일상 관리 (Priority: P1)

사용자가 메인 화면에서 오늘의 할 일을 추가, 조회, 수정, 비활성화, 삭제할 수 있다. 시간대에 따라 계획(Plan) 모드 또는 회고(Review) 모드가 자동 전환되며, 수동 전환도 가능하다. 할 일의 상태(활성/비활성/완료)를 직관적인 제스처로 관리한다.

**Why this priority**: 할 일 관리는 앱의 핵심 가치이며, 이 기능 없이는 서비스의 존재 이유가 없다.

**Independent Test**: 할 일 추가 → 조회 → 수정 → 완료/비활성화 → 삭제 흐름을 테스트할 수 있으며, 이것만으로 기본적인 할 일 관리 가치를 전달한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 메인 화면에 진입하면, **When** 현재 시간이 계획 시간대이면, **Then** Plan 모드가 자동으로 표시되고, 회고 시간대이면 Review 모드가 표시된다.
2. **Given** Plan 모드에서, **When** "+" 버튼을 탭하고 할 일 내용을 입력하면, **Then** 오늘 목록에 즉시 추가된다.
3. **Given** 목록에 할 일이 있을 때, **When** 해당 항목을 탭하여 내용을 수정하고 다른 영역을 탭하면, **Then** 변경된 내용이 자동 저장된다.
4. **Given** 목록에 할 일이 있을 때, **When** 체크박스를 탭하면, **Then** 해당 항목이 완료 상태로 변경되고 시각적으로 구분된다.
5. **Given** 목록에 할 일이 있을 때, **When** 더블 탭하면, **Then** 취소선이 표시되고 비활성화(이월 대상 제외) 상태가 된다.
6. **Given** 목록에 할 일이 있을 때, **When** 길게 눌러 하단 휴지통으로 드래그하면, **Then** 해당 항목이 삭제된다.
7. **Given** 사용자가 모드 전환을 원할 때, **When** 수동으로 모드를 전환하면, **Then** Plan/Review 모드가 즉시 전환된다.

### User Story 3 - 하루 회고 및 자동 이월 (Priority: P1)

사용자가 하루 끝에 회고 모드에서 오늘의 진행률을 확인하고, 완료 여부를 최종 체크한 뒤 "오늘의 일정 완료" 버튼으로 하루를 마감한다. 미완료 항목은 자동으로 다음 날로 이월된다. 자정이 지나면 별도 조작 없이도 미완료 항목이 다음 날 목록에 자동 표시된다.

**Why this priority**: "계획 → 실행 → 회고" 루틴의 마지막 단계이며, 자동 이월은 사용자가 중요한 할 일을 놓치지 않게 하는 핵심 차별점이다.

**Independent Test**: 회고 모드 진입 → 진행률 확인 → 일정 완료 → 이월 확인 흐름을 테스트할 수 있으며, 루틴 완성이라는 가치를 전달한다.

**Acceptance Scenarios**:

1. **Given** Review 모드에 진입했을 때, **When** 오늘의 할 일 목록이 표시되면, **Then** 완료/미완료 항목이 분리 표시되고 진행률(완료/전체)이 보인다.
2. **Given** Review 모드에서 미처 체크하지 못한 항목이 있을 때, **When** 완료 상태를 변경하면, **Then** 진행률이 즉시 재계산된다.
3. **Given** Review 모드에서 정리를 마친 상태에서, **When** "오늘의 일정 완료" 버튼을 탭하면, **Then** 기록이 저장되고 미완료(ACTIVE) 항목이 다음 날로 이월된다.
4. **Given** 어제 미완료 항목이 존재할 때, **When** 자정 이후 앱에 접속하면, **Then** 이월된 항목이 오늘 목록에 자동으로 표시된다.
5. **Given** 이미 완료 처리된 날짜에 대해, **When** 다시 완료 버튼을 누르면, **Then** 중복 완료가 방지된다.

### User Story 4 - 푸시 알림 (Priority: P2)

사용자가 설정한 계획 시간과 회고 시간에 맞춰 푸시 알림을 받아 루틴을 놓치지 않도록 한다. 알림 시간이 설정되지 않은 경우(NULL) 해당 알림은 발송되지 않는다.

**Why this priority**: 루틴 형성의 핵심 트리거이지만, 알림 없이도 앱을 직접 열어 사용할 수 있으므로 핵심 CRUD보다 후순위이다.

**Independent Test**: 계획/회고 시간 설정 후 해당 시간에 알림 수신 여부를 테스트할 수 있으며, 루틴 리마인더라는 가치를 전달한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 계획 알림 시간을 08:00으로 설정한 상태에서, **When** 08:00이 도래하면, **Then** 계획 시작을 유도하는 푸시 알림이 발송된다.
2. **Given** 사용자가 회고 알림 시간을 22:00으로 설정한 상태에서, **When** 22:00이 도래하면, **Then** 회고를 유도하는 푸시 알림이 발송된다.
3. **Given** 사용자가 계획 알림 시간을 NULL로 설정한 상태에서, **When** 어떤 시간이든, **Then** 계획 알림이 발송되지 않는다.
4. **Given** 사용자가 설정 화면에서 알림 시간을 변경했을 때, **When** 새로운 시간이 도래하면, **Then** 변경된 시간에 맞춰 알림이 발송된다.

### User Story 5 - 메모 첨부 (Priority: P2)

사용자가 개별 할 일 항목에 메모를 추가, 수정, 삭제하여 상세 정보를 기록할 수 있다.

**Why this priority**: 할 일의 부가 정보를 관리하는 편의 기능으로, 핵심 CRUD 없이는 의미가 없다.

**Independent Test**: 할 일에 메모 추가 → 수정 → 삭제 흐름을 테스트할 수 있으며, 상세 기록이라는 가치를 전달한다.

**Acceptance Scenarios**:

1. **Given** 할 일 항목이 선택된 상태에서, **When** 메모를 작성하고 저장하면, **Then** 해당 할 일에 메모가 첨부되어 이후에도 확인할 수 있다.
2. **Given** 메모가 첨부된 할 일에서, **When** 메모 내용을 수정하면, **Then** 변경된 내용이 즉시 반영된다.
3. **Given** 메모가 첨부된 할 일에서, **When** 메모를 삭제하면, **Then** 메모가 제거된다.

---

### User Story 6 - 음성 인식 할 일 추가 (Priority: P3)

사용자가 마이크 버튼을 탭하여 음성으로 할 일을 입력하면, STT(음성→텍스트)와 LLM이 음성을 다듬어 깔끔한 할 일 텍스트로 변환하여 오늘 목록에 추가한다.

**Why this priority**: 편의 기능으로, 텍스트 입력 방식이 이미 존재하므로 핵심보다 후순위이다.

**Independent Test**: 마이크 버튼 탭 → 음성 입력 → 변환된 텍스트 확인 → 목록 추가 흐름을 테스트할 수 있다.

**Acceptance Scenarios**:

1. **Given** 계획 화면에서, **When** 마이크 버튼을 탭하고 "오후 3시에 팀 미팅 가야 되는데"라고 말하면, **Then** LLM이 "오후 3시 팀 미팅 참석"으로 다듬어 오늘 목록에 추가된다.
2. **Given** 음성 입력 중, **When** 음성 인식에 실패하면, **Then** 사용자에게 재시도 안내 메시지가 표시된다.

---

### User Story 7 - 캘린더 조회 (Priority: P3)

사용자가 캘린더 페이지에서 날짜별로 할 일을 조회하고, 월별 요약(완료/전체 건수)을 확인한다.

**Why this priority**: 과거 및 미래 일정을 확인하는 부가 기능으로, 오늘의 할 일 관리가 핵심이다.

**Independent Test**: 캘린더 페이지 진입 → 월별 요약 확인 → 특정 날짜 선택 → 할 일 목록 확인 흐름을 테스트할 수 있다.

**Acceptance Scenarios**:

1. **Given** 캘린더 페이지에 진입했을 때, **When** 월별 보기가 표시되면, **Then** 각 날짜의 할 일 건수와 완료 건수가 요약 표시된다.
2. **Given** 캘린더에서, **When** 특정 날짜를 탭하면, **Then** 해당 날짜에 등록된 할 일 목록이 표시된다.

---

### User Story 8 - TodoList 공유 (Priority: P4)

사용자가 자신의 할 일 목록을 SNS/메신저로 공유하여 하루 마무리의 성취감을 나눌 수 있다. 공유 목록 최상단에는 "나에게 전송" 옵션이 배치된다.

**Why this priority**: 소셜 기능으로, 개인 일정 관리가 우선이다.

**Independent Test**: 날짜 선택 → 공유 버튼 탭 → 공유 채널 선택 흐름을 테스트할 수 있다.

**Acceptance Scenarios**:

1. **Given** 공유할 일정이 존재할 때, **When** 날짜를 선택하고 공유 버튼을 탭하면, **Then** "나에게 전송"이 최상단에 표시되고 그 아래에 SNS/메신저 채널이 나열된다.

### Edge Cases

- 사용자의 타임존이 변경된 경우 계획/회고 시간과 자정 기준 이월 처리가 새 타임존에 맞게 동작해야 한다.
- 자정 이월 시점에 앱이 실행 중인 경우와 종료된 경우 모두 이월이 정상 동작해야 한다.
- 하루에 할 일이 0개인 상태에서 "오늘의 일정 완료"를 누를 경우 정상 처리되어야 한다.
- OAuth 인증 도중 사용자가 취소하거나 provider에서 에러가 반환된 경우 적절한 안내가 표시되어야 한다.
- 이월된 할 일이 다시 미완료로 남을 경우 연속 이월이 정상 동작해야 한다.
- 음성 인식 시 지원하지 않는 오디오 포맷이 업로드된 경우 명확한 에러 메시지가 표시되어야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 OAuth 2.0 소셜 로그인(Google, Naver, Kakao, Apple)을 통해 사용자를 인증해야 한다.
- **FR-002**: 시스템은 신규 사용자 로그인 시 자동으로 계정을 생성해야 한다.
- **FR-003**: 시스템은 JWT 기반 Access Token과 Refresh Token을 발급하고, 만료 시 자동 갱신해야 한다.
- **FR-004**: 시스템은 로그아웃 시 세션 토큰을 무효화하고 FCM 디바이스 토큰을 삭제해야 한다.
- **FR-005**: 신규 사용자는 최초 로그인 시 온보딩 화면에서 계획 시간(기본 08:00)과 회고 시간(기본 22:00)을 설정해야 하며, 설정 완료 전에는 메인 화면에 진입할 수 없다.
- **FR-006**: 사용자는 설정 화면에서 언제든지 계획 시간, 회고 시간, 타임존, 언어를 변경할 수 있어야 한다. 시간을 NULL로 설정하면 해당 알림이 해제된다.
- **FR-007**: 메인 화면은 사용자의 계획 시간과 회고 시간을 기준으로 Plan/Review 모드가 자동 전환되어야 하며, 수동 전환도 가능해야 한다.
- **FR-008**: 사용자는 오늘 날짜의 할 일을 추가할 수 있어야 한다(최대 255자).
- **FR-009**: 사용자는 할 일 항목을 탭하여 내용을 수정하고, 다른 영역 탭 시 자동 저장되어야 한다.
- **FR-010**: 사용자는 체크박스를 통해 할 일을 완료 상태로 변경할 수 있어야 한다.
- **FR-011**: 사용자는 더블 탭으로 할 일을 비활성화(취소선 표시, 이월 대상 제외)할 수 있어야 한다.
- **FR-012**: 사용자는 길게 누르기 + 드래그로 할 일을 삭제할 수 있어야 한다(하단 휴지통 UI 표시).
- **FR-013**: 시스템은 특정 날짜의 할 일 목록을 이월 항목 포함하여 조회할 수 있어야 한다.
- **FR-014**: 시스템은 사용자가 설정한 계획/회고 시간에 푸시 알림을 발송해야 한다. 시간이 NULL인 경우 해당 알림을 발송하지 않아야 한다.
- **FR-015**: 시스템은 알림 발송 시 사용자의 타임존을 고려해야 한다.
- **FR-016**: Review 모드에서 오늘의 진행률(완료/전체)과 완료/미완료 항목 분리 요약이 표시되어야 한다.
- **FR-017**: 사용자는 "오늘의 일정 완료" 버튼으로 하루를 마감하고, 미완료(ACTIVE) 항목을 다음 날로 자동 이월할 수 있어야 한다.
- **FR-018**: 시스템은 자정(사용자 타임존 기준)에 미완료 항목을 자동으로 다음 날로 이월해야 한다.
- **FR-019**: 사용자는 개별 할 일에 메모를 추가, 수정, 삭제할 수 있어야 한다.
- **FR-020**: 사용자는 마이크 버튼을 통해 음성으로 할 일을 입력할 수 있어야 하며, STT와 LLM이 음성을 다듬어 텍스트로 변환해야 한다.
- **FR-021**: 캘린더 페이지에서 날짜별 할 일과 월별 요약(완료/전체 건수)을 조회할 수 있어야 한다.
- **FR-022**: 사용자는 할 일 목록을 SNS/메신저로 공유할 수 있어야 하며, "나에게 전송" 옵션이 최상단에 배치되어야 한다.
- **FR-023**: 시스템은 로그인 시 FCM 디바이스 토큰을 등록하여 푸시 알림 수신을 가능하게 해야 한다.
- **FR-024**: 모든 인터랙티브 UI 요소(버튼, 입력 필드, 체크박스, 탭 가능 항목 등)에 고유한 `testID` 속성을 부여해야 한다. Maestro E2E 테스트에서 testID 기반 요소 탐색을 우선하며, testID가 없는 경우에만 텍스트로 fallback한다.
- **FR-025**: P1-P2(US1~US5) 기능은 각 구현 Phase 완료 시 Maestro E2E 테스트를 작성해야 한다. P3-P4(US6~US8) 기능은 Polish 이전 별도 Phase에서 외부 API key 등록과 함께 E2E 테스트를 작성해야 한다.
- **FR-026**: 모든 E2E 테스트는 `.maestro/` 디렉토리에 YAML로 작성하며, appId는 `com.todolist.app`을 사용한다. 각 플로우는 `launchApp` + `clearState: true`로 시작하고, 한 플로우당 하나의 유저 저니만 테스트한다.
- **FR-027**: 코드 내 미해결 사항은 `TODO(담당자): #이슈번호 설명` 형식으로 작성하고, GitHub Issues에 대응하는 이슈를 등록해야 한다.

### Key Entities

- **User**: 서비스 사용자를 나타낸다. 이름, 계획 시간, 회고 시간, 타임존, 언어 설정을 가진다.
- **User Auth**: 사용자의 인증 정보를 나타낸다. 로그인 ID와 비밀번호 해시를 가진다.
- **User Auth OAuth**: OAuth provider별 연동 정보를 나타낸다. provider 종류, provider 사용자 ID, 이메일을 가진다. User Auth와 연관된다.
- **User Session**: 사용자의 로그인 세션을 나타낸다. Refresh Token, 접속 정보, 만료 시간을 가진다.
- **Todo**: 할 일 항목을 나타낸다. 내용, 날짜, 상태(ACTIVE/INACTIVE/COMPLETED/CARRIED_OVER)를 가진다. User에 소속된다.
- **Carried Over History**: 이월 이력을 나타낸다. 원본 할 일과 이월된 할 일의 관계를 추적한다.
- **Todo Memo**: 할 일에 첨부된 메모를 나타낸다. 하나의 할 일에 여러 메모가 첨부될 수 있다.
- **Notification Log**: 알림 발송 이력을 나타낸다. 알림 유형(PLAN/REVIEW), 발송 상태(SUCCESS/FAIL), 재시도 횟수를 가진다.
- **User Device**: 사용자의 디바이스를 나타낸다. FCM 토큰, 디바이스 유형(IOS/ANDROID), 디바이스 이름을 가진다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 소셜 로그인부터 온보딩 완료까지 1분 이내에 완료할 수 있다.
- **SC-002**: 앱 실행 후 메인 화면 로딩까지 2초 이내에 완료된다.
- **SC-003**: 할 일 추가/수정/삭제 조작 후 1초 이내에 화면에 반영된다.
- **SC-004**: 알림이 사용자가 설정한 시간 대비 1분 이내 오차로 발송된다.
- **SC-005**: 서비스 가용률이 99.5% 이상 유지된다.
- **SC-006**: 90% 이상의 신규 사용자가 온보딩을 완료하고 첫 할 일을 추가한다.
- **SC-007**: 이월 대상 미완료 항목의 100%가 다음 날 목록에 정상 표시된다.
- **SC-008**: iOS 15+ 및 Android 12+ 환경에서 모든 기능이 정상 동작한다.

## Assumptions

- 사용자는 인터넷에 연결된 모바일 디바이스(iOS 또는 Android)를 사용한다.
- OAuth provider(Google, Naver, Kakao, Apple)의 인증 서비스가 정상 운영 중이다.
- 외부 STT 및 LLM API 서비스가 정상 운영 중이며, 한국어를 지원한다.
- FCM(Firebase Cloud Messaging) 서비스가 정상 운영 중이다.
- 사용자의 디바이스에 푸시 알림 권한이 허용되어 있어야 알림을 수신할 수 있다.
- 오프라인 모드는 v1 범위에 포함하지 않으며, 모든 조작에 네트워크 연결이 필요하다.
- 공유 기능은 OS 기본 공유 시트를 활용하며, 별도의 딥링크/URL 생성 서버는 v1에서 구현하지 않는다.
- 데이터 암호화는 전송 중 TLS, 저장 시 Supabase 기본 디스크 암호화(AES-256)를 적용한다. 추가 애플리케이션 레벨 암호화는 v1에서 적용하지 않는다.
- v1에서는 기본 접근성만 적용한다 (접근성 라벨, 최소 색상 대비 4.5:1). WCAG 2.1 AA 완전 준수는 v2 범위이다.
- 모든 코드 변경은 `feature/*` 브랜치에서 작업하며, main 브랜치로의 병합은 PR(Pull Request) 리뷰를 필수로 거친다 (헌법 IX조).
- 코드 주석은 WHY 중심으로 작성하고, 공개 API는 JSDoc/TSDoc으로 문서화한다. 주석 처리된 코드와 코드 반복 주석은 금지한다 (헌법 VIII조).
- 이슈 트래킹은 GitHub Issues를 사용하며, TODO/FIXME 주석은 `TODO(담당자): #이슈번호 설명` 형식을 따른다 (헌법 VIII조).

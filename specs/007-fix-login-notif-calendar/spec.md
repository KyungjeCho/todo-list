## Clarifications

### Session 2026-04-14

- Q: Group B(FCM) 대상 플랫폼과 테스트 방침은? → A: iOS/Android **양쪽 모두 FCM 도달**이 요구사항. iOS 실기기 테스트가 현재 불가하므로 **iOS 검증은 Android 실기기 테스트로 대체**(대리 검증)하며, iOS 전용 동작은 스펙상 요구사항으로 유지하되 측정 SC는 Android 기준으로 선언.
- Q: iOS FCM/APNs 사전 구성(Apple Developer Program 등록, APNs 인증서, Firebase iOS 앱 등록, 번들 구성)의 현재 상태와 본 스펙 포함 여부는? → A: **Apple Developer Program 미등록** 상태로, APNs 인증서 발급 자체가 불가. 따라서 iOS 실제 알림 도달은 외부 선행 작업(개발자 프로그램 가입 → APNs 인증서 → Firebase iOS 앱 등록 → 번들 구성)이 완료되기 전까지 **불가능**하며, 해당 인프라 작업은 **본 스펙 범위 밖**이다. 본 스펙은 iOS 지원을 "코드 레벨 준비(플랫폼 공통 권한 요청·토큰 처리 경로 유지)"로 한정하고, **실제 iOS 전송·수신 검증은 후속 이터레이션**(개발자 프로그램 등록 이후)으로 이관한다.
- Q: Apple Developer Program 등록을 이번 이터레이션 범위에 포함해야 하는가? → A: **No — 명시적 Out of Scope**. 본 이터레이션에서는 어떠한 Apple Developer 계정 가입·결제·인증서 발급·프로비저닝 프로파일 작업도 수행하지 않는다. iOS 실동작 검증과 APNs/Firebase iOS 등록·번들 구성 전체가 이 결정에 의해 자동으로 Out of Scope로 귀속된다(아래 "Out of Scope" 섹션 참조).

## Out of Scope

다음 항목들은 **본 이터레이션(007) 범위에서 명시적으로 제외**된다. 각 항목은 요구사항이 아니며, 본 feature 완료 판정 시 충족 여부를 따지지 않는다. 별도 후속 이터레이션에서 다룬다.

- **Apple Developer Program 등록** (계정 가입, 연간 결제, 팀 설정) — 본 이터레이션에서 수행하지 않는다.
- **APNs 인증서 발급·갱신** 및 Firebase iOS 앱 등록, iOS 번들(.plist/엔타이틀먼트/프로비저닝 프로파일) 구성.
- **iOS 실기기 실동작 검증** — 알림 수신·표시·백그라운드 동작 등 iOS 전용 동작의 측정·수락 판정.
- **iOS 전용 SC 측정** — SC-005의 iOS 수치 측정(≥ 95% 도달률 등).
- **iOS 스토어 배포** 관련 작업(TestFlight 포함).

본 이터레이션에서 iOS 관련 범위는 **오직 코드 레벨 준비(플랫폼 공통 권한 요청·토큰 처리·수신 핸들러 경로의 방어적 유지)** 와 **APNs 미구성 상태에서의 앱 안정성(크래시 없음)** 까지로 한정된다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 온보딩을 마친 사용자가 알림을 꺼도 메인 화면에 머문다 (Priority: P1)

온보딩(초기 설정)을 이미 완료한 사용자가 설정 화면에서 계획/회고 알림을 OFF로 변경해도 앱은 메인 화면에 그대로 머문다. 로그인 직후에도 화면이 메인 → 온보딩 → 메인으로 깜빡이지 않고, 최종 목적지 화면이 한 번에 표시된다. 앞으로 알림 설정이 어떻게 바뀌든 "온보딩을 통과했다"는 사실은 독립적으로 기록·유지된다.

**Why this priority**: 4건의 결함(이슈 1, 3, 4, 5)이 하나의 구조적 원인(온보딩 완료 여부를 알림 시간 필드 존재 여부로 간접 추론)에서 파생되므로, 이 그룹을 먼저 해결해야 현재 사용자가 겪는 "알림 OFF 시 온보딩 재진입", "로그인 후 화면 플리커" 같은 가장 눈에 띄는 UX 저하가 사라진다. 재가입 없이 기존 사용자 전원이 즉시 혜택을 본다.

**Independent Test**: 온보딩을 완료한 테스트 계정으로 로그인한 뒤 (1) 플리커 없이 메인이 1회 렌더되는지, (2) 설정에서 모든 알림을 OFF 한 뒤에도 메인에 유지되는지, (3) 앱을 완전히 재시작해도 여전히 메인으로 진입하는지 확인한다. 온보딩 미완료 계정은 온보딩 화면으로 정상 진입해야 한다.

**Acceptance Scenarios**:

1. **Given** 온보딩을 마친 기존 사용자가 로그인 중이다, **When** 프로필 로딩이 끝나고 최초 화면이 그려진다, **Then** 사용자는 메인 화면을 단 한 번만 보고 온보딩 화면 전환 플리커는 발생하지 않는다.
2. **Given** 온보딩을 마친 사용자가 메인 화면에 있다, **When** 설정에서 계획 알림과 회고 알림을 모두 OFF 한다, **Then** 온보딩 화면으로 이동하지 않고 메인 화면에 머문다.
3. **Given** 온보딩을 아직 마치지 않은 신규 사용자가 로그인한다, **When** 홈 진입 판정이 끝난다, **Then** 사용자는 온보딩 화면으로 이동한다.
4. **Given** 온보딩을 마친 사용자가 임의의 경로로 온보딩 화면에 접근하려 한다, **When** 해당 경로를 따라 이동한다, **Then** 가드가 작동하여 온보딩 화면 진입이 차단되고 메인 화면으로 돌아간다.
5. **Given** 온보딩 완료 상태를 기록하는 명시적 동작이 호출된다, **When** 이미 완료한 사용자가 같은 동작을 다시 호출한다, **Then** 시스템은 에러 없이 현재 프로필을 반환한다(멱등).
6. **Given** 본 기능 배포 시점에 이미 앱을 사용 중이던 모든 기존 사용자, **When** 배포 후 다음 로그인 혹은 프로필 재조회가 일어난다, **Then** 재온보딩 요구 없이 메인 화면으로 진입한다.

### User Story 2 - iOS/Android 개발 빌드에서 예약 알림이 실제로 표시된다 (Priority: P2)

개발 빌드를 실기기에 설치한 사용자가 백엔드 스케줄러 또는 수동 트리거로 발송된 계획/회고 알림을 알림 센터에서 실제로 볼 수 있다. 플랫폼별 최초 실행 시 알림 권한 프롬프트가 표시되고, 사용자가 허용하면 이후 알림이 정상 수신된다. iOS/Android **양쪽 모두**가 스펙상 요구사항이나, 본 이터레이션에서 **iOS 실기기 테스트는 불가**하므로 검증은 **Android 실기기**로 대체한다.

**Why this priority**: 기능 자체는 존재하지만 **결과가 전혀 도달하지 않는** 결함으로, 개발·QA 진행을 막는 블로커 성격이다. 다만 수정은 권한/채널 설정에 국한되고 Group A 만큼 광범위한 사용자 영향(라우팅 오판)은 아니므로 P2.

**Independent Test**: Android 13 이상 실기기에 개발 빌드를 새로 설치 → 최초 실행 시 알림 권한 프롬프트가 뜨는지 확인 → 허용 → 백엔드에서 예약/수동 트리거로 알림 발송 → 기기 알림 센터에 표시되는지 확인. (Group A/C 진행 여부와 무관하게 단독 검증 가능.) iOS 동일 시나리오는 실기기 부재로 **본 이터레이션에서 측정 불가** — Android 결과를 대리 근거로 삼고, iOS 실기기 확보 시 동일 절차로 회귀 검증한다.

**Acceptance Scenarios**:

1. **Given** Android 13 이상 실기기에 개발 빌드를 처음 설치한다, **When** 앱을 최초 실행한다, **Then** Android 알림 권한 프롬프트가 표시된다.
2. **Given** Android 사용자가 알림 권한을 허용한 상태다, **When** 백엔드가 해당 사용자에게 푸시 알림을 발송한다, **Then** 기기의 알림 센터에 알림이 표시된다.
3. **Given** 사용자가 알림 권한을 거부했다, **When** 백엔드가 해당 사용자에게 푸시 알림을 발송한다, **Then** 알림은 표시되지 않지만 앱은 충돌 없이 정상 동작한다(플랫폼 공통).
4. **Given** Android 앱이 백그라운드 또는 종료 상태다, **When** 예약된 알림 시각이 도래한다, **Then** 알림 센터에 알림이 표시된다.
5. **Given** iOS 실기기 테스트가 불가하고 Apple Developer 미등록으로 APNs 구성도 불가한 현재 상황이다, **When** 본 이터레이션의 수락 검증을 수행한다, **Then** 이 US2의 실동작 수락 기준은 Android 한정으로 측정하며, iOS 전용 실동작 검증(도달·표시)은 외부 선행 작업(Apple Developer 등록 → APNs 구성) 완료 후 회귀 이터레이션으로 이관된다.
6. **Given** iOS 개발 빌드(APNs 미구성)가 실행된다, **When** 앱이 알림 초기화 경로를 통과한다, **Then** 크래시·치명적 에러 없이 방어적 no-op으로 동작하며 Group A/C 기능은 영향을 받지 않는다.

### User Story 3 - 새 할 일을 추가하면 캘린더가 즉시 최신 상태를 반영한다 (Priority: P3)

홈 탭에서 할 일을 추가한 직후 캘린더 탭으로 전환하면, 새로 추가한 할 일이 해당 날짜의 월간 요약에 반영되어 표시된다. 홈 탭 최초 추가 이후라도 캘린더 탭이 이전 달 혹은 과거 시점의 집계를 보여주지 않는다.

**Why this priority**: 데이터 일관성 문제이기는 하나, 사용자가 탭을 다시 열 때 자동 새로고침만 되어도 충분히 해소된다. Group A(라우팅 오판)나 Group B(알림 미도달)만큼 기능 실패가 아니며, 수정 범위가 가장 작다.

**Independent Test**: 로그인 후 홈 탭에서 오늘 날짜에 할 일을 1개 추가 → 캘린더 탭으로 전환 → 오늘 날짜의 월간 요약 카운트가 증가했는지 확인. 추가 전/후 캘린더 월간 요약 수치의 차이만 검증하면 된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 홈 탭에서 오늘 날짜에 할 일을 추가했다, **When** 캘린더 탭으로 전환한다, **Then** 해당 날짜의 월간 요약에 방금 추가한 항목이 반영되어 표시된다.
2. **Given** 사용자가 캘린더 탭에 머물고 있다, **When** 표시되는 월(year/month)이 바뀐다, **Then** 새로운 월에 대한 최신 요약이 표시된다.
3. **Given** 사용자가 캘린더 탭을 떠났다가 돌아온다, **When** 캘린더 탭이 다시 포커스를 얻는다, **Then** 현재 월의 최신 요약이 다시 로드되어 표시된다.

### Edge Cases

- 기존 사용자 마이그레이션 시, 일부 사용자가 실제로는 온보딩 화면을 끝까지 거치지 않았더라도 **일괄 완료 처리**되어 온보딩 재진입이 요구되지 않는다(의도된 트레이드오프, 마이그레이션 승인 사항).
- 사용자가 신규 가입 후 온보딩 완료 기록이 남기 전에 앱을 강제 종료한다 → 다음 로그인 시 여전히 온보딩 화면으로 진입한다.
- 프로필 로딩이 네트워크 지연으로 길어지는 경우 사용자는 메인/온보딩 어느 쪽도 보지 않고 로딩 상태만 관찰한다(플리커 방지).
- Android 13 미만 기기에서는 알림 권한 프롬프트가 표시되지 않고, 별도 권한 없이 알림이 동작한다.
- iOS 빌드는 Apple Developer 미등록·APNs 미구성 상태에서 실행되므로 푸시 도달은 발생하지 않는다. 알림 초기화 경로는 크래시 없이 방어적 no-op으로 동작하며, 사용자에게 노출되는 알림 관련 UI(설정 화면 등)는 평소와 동일하게 렌더된다(iOS 전용 검증은 후속 이터레이션으로 이관).
- 사용자가 홈 탭에서 할 일을 동일한 월에 여러 번 추가한 뒤 캘린더로 전환 → 마지막 추가분까지 포함된 요약이 표시된다.
- 사용자가 홈 탭 추가 직후 네트워크가 끊긴 상태에서 캘린더로 전환 → 기존 캐시 유지 + 재조회 실패는 사용자에게 거슬리지 않게 기존 값 유지(에러 배너는 선택).

## Requirements *(mandatory)*

### Functional Requirements

#### Group A — 온보딩 완료 상태 (이슈 1, 3, 4, 5)

- **FR-001**: 시스템은 사용자별로 "온보딩 완료 여부"를 알림 시간(계획/회고) 설정과 **분리된 독립 상태**로 보관해야 한다.
- **FR-002**: 시스템은 사용자가 알림을 OFF로 변경하더라도 온보딩 완료 상태를 변경하지 않아야 한다.
- **FR-003**: 시스템은 로그인 직후 프로필 로딩이 끝나기 전까지 메인 화면도 온보딩 화면도 렌더하지 않고 로딩 상태를 유지해야 한다(플리커 방지).
- **FR-004**: 시스템은 로그인 완료 직후 온보딩 완료 여부에 따라 정확히 **한 번** 목적 화면(메인 또는 온보딩)으로 전환해야 한다.
- **FR-005**: 시스템은 "온보딩 완료" 상태 전이를 기록하는 **명시적 도메인 액션**을 제공해야 하며, 이 액션은 시간 설정 여부와 무관하게 호출 가능해야 한다.
- **FR-006**: FR-005의 액션은 **멱등**해야 한다(이미 완료 상태에서 재호출되어도 에러 없이 현재 프로필을 반환).
- **FR-007**: 시스템은 온보딩 완료 사용자가 온보딩 화면에 접근하는 것을 가드하고, 메인 화면으로 리다이렉트해야 한다. 구현 방식은 `AuthNavigator` 에서 `showOnboarding` 이 false 인 동안 Onboarding 스택의 `<Stack.Screen>` 자체를 렌더하지 않는 **구조적 가드**로 고정한다(명령형 `navigation.reset` 후처리가 아니라 라우트 정의 자체를 조건부로 제거하여 딥링크·복원·뒤로가기를 포함한 모든 경로에서 일관되게 차단).
- **FR-008**: 시스템은 본 기능 배포 시점에 이미 존재하는 모든 사용자를 **온보딩 완료 상태로 일괄 백필**하여 재온보딩이 요구되지 않도록 해야 한다.
- **FR-009**: 시스템은 신규 가입 사용자의 기본 온보딩 완료 상태를 **미완료**로 시작해야 한다.

#### Group B — iOS/Android 알림 수신 (이슈 2)

- **FR-010**: 시스템은 플랫폼별 규약에 따라 앱 최초 실행 시 알림 권한을 요청해야 한다(Android 13 이상: 런타임 권한 프롬프트, iOS: 표준 알림 권한 프롬프트). iOS 동작 경로는 **코드 레벨로 준비**되어 있어야 하며, 실제 동작은 Apple Developer/APNs 구성 완료 후 활성화된다.
- **FR-011**: 시스템은 사용자가 알림 권한을 허용하면 **Android**에서 백엔드가 발송한 푸시 알림을 기기의 알림 센터에 표시해야 한다. **iOS**는 동일한 결과가 요구사항이나, **Apple Developer Program 등록 및 APNs 구성 완료 후**에만 실제 검증이 가능하며 해당 인프라 작업은 본 스펙 범위 밖이다.
- **FR-012**: 시스템은 앱이 백그라운드/종료 상태에서도 푸시 알림을 수신·표시해야 한다(Android 필수, iOS는 FR-011 단서와 동일 조건). 구현은 앱 엔트리(`frontend/index.ts`)에서 `messaging().setBackgroundMessageHandler` 를 `registerRootComponent` **이전**에 등록하여 헤드리스 JS 실행 경로를 확보한다(훅 수명주기로는 종료 상태 수신이 보장되지 않으므로 엔트리 등록이 필수).
- **FR-013**: 시스템은 사용자가 알림 권한을 거부한 경우에도 충돌 없이 동작하고, 권한이 없음을 전제로 이후 플로우를 진행해야 한다(iOS/Android 공통 — 권한 거부 처리는 인프라와 무관).
- **FR-013a**: 시스템의 iOS 측 알림 관련 코드(권한 요청 호출, 토큰 획득·업로드 경로, 수신 핸들러)는 APNs 미구성 상태에서도 **앱 충돌이나 기능 저해 없이 안전하게 no-op 또는 방어적으로 동작**해야 한다.

#### Group C — 캘린더 즉시 반영 (이슈 6)

- **FR-014**: 시스템은 사용자가 할 일을 추가·수정·삭제한 직후 캘린더 탭에서 해당 변경이 월간 요약에 반영된 상태로 관측 가능해야 한다(적어도 탭 포커스 전환 시점까지).
- **FR-015**: 시스템은 캘린더 탭이 다시 포커스를 얻을 때마다 현재 표시 중인 월의 요약을 최신 값으로 갱신해야 한다.
- **FR-016**: 시스템은 캘린더 탭에서 다른 월로 이동할 때 해당 월의 요약을 최신 값으로 불러와야 한다.

### Key Entities *(include if feature involves data)*

- **사용자 프로필(User Profile)**: 사용자의 기본 설정과 상태를 담는 단위. 본 기능으로 "온보딩 완료 여부"라는 속성이 **알림 시간과 독립된 1급 속성**으로 추가된다. 기존 속성(계획 알림 시각, 회고 알림 시각, 시간대, 언어)은 유지되며 "온보딩 완료 여부"와의 인과 관계는 끊어진다.
- **할 일(Todo)**: 사용자가 생성·수정·삭제하는 작업 항목. 데이터 구조 변경은 없으나, 변경 이벤트가 캘린더 뷰의 월간 요약에 반영되어야 한다는 요구(FR-014)를 발생시킨다.
- **월간 요약(Monthly Summary)**: 특정 연/월의 할 일 통계 뷰. 데이터 구조 변경 없음. 갱신 트리거(포커스, 월 변경)만 요구사항으로 명시된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 온보딩을 완료한 사용자가 설정에서 모든 알림을 OFF 로 변경하는 시나리오에서, 온보딩 화면으로 되돌아가는 사용자는 **0명**이다(회귀 테스트 100% 통과).
- **SC-002**: 온보딩을 완료한 사용자의 **로그인 직후 화면 전환 횟수는 1회**이며, 메인↔온보딩 플리커 발생률은 **0%** 다.
- **SC-003**: 본 기능 배포 직후 기존 사용자 중 **재온보딩이 요구되는 사용자 수는 0명**이다.
- **SC-004**: 신규 가입 사용자가 온보딩을 완료하기까지 걸리는 추가 단계는 기존 대비 **증가하지 않는다**(백엔드에 완료 기록이 남는 한 단계만 추가되며 사용자 체감 단계는 그대로).
- **SC-005**: Android 13 이상 실기기 개발 빌드에서 허용된 사용자 기준, 발송된 알림이 알림 센터에 표시되는 비율은 **95% 이상**이다(네트워크 정상 조건). iOS는 본 이터레이션에서 (1) 실기기 확보 불가 + (2) Apple Developer 미등록으로 APNs 발급 불가 — 두 제약으로 **수치 측정 및 실동작 검증에서 제외**되며, Android 결과를 플랫폼 공통 구현부의 대리 지표로 사용한다. iOS 전용 측정은 Apple Developer 등록·APNs 발급 및 실기기 확보 이후 별도 이터레이션에서 동일 기준(≥ 95%)으로 검증한다.
- **SC-008**: 본 이터레이션 완료 시점에 iOS 빌드는 Apple Developer/APNs 미구성 상태에서도 **앱 충돌 없이 정상 구동**되며, 알림 관련 코드 경로는 방어적 no-op으로 동작한다(크래시 리포트 0건, 관련 에러 로그가 앱 기능을 차단하지 않음).
- **SC-006**: 홈 탭에서 할 일을 추가한 뒤 캘린더 탭으로 전환했을 때 월간 요약이 최신 값으로 표시되는 비율은 **100%** 다(수동/자동 테스트 기준).
- **SC-007**: 관련 사용자 보고(온보딩 오진입, 알림 미수신, 캘린더 미갱신) 티켓이 배포 후 1개월간 **0건** 이다.

## Assumptions

- 기존 사용자의 온보딩 이력은 정확한 복원이 어려우므로 **전원 완료 처리(백필)** 한다는 운영상의 트레이드오프가 이미 승인되어 있다. 일부 사용자가 실제로는 온보딩을 중단했더라도 재진입을 요구하지 않는다.
- 개발·QA 환경은 **Android 13(API 33) 이상 실기기**를 포함한다. iOS는 요구사항상 알림 수신 대상이나 (1) 실기기 확보 불가, (2) **Apple Developer Program 등록이 본 이터레이션 Out of Scope** 로 확정되어 APNs 발급이 불가 — 두 제약이 동시에 존재한다. 따라서 본 이터레이션에서 iOS는 **코드 레벨 준비**까지만 수행하고, 실제 iOS 전송·수신 검증과 APNs/Firebase iOS 등록 작업은 **후속 이터레이션(개발자 프로그램 등록 이후)** 에서 진행한다(상세는 "Out of Scope" 섹션). Android 결과는 플랫폼 공통 구현부의 대리 근거로 사용한다.
- 백엔드 알림 스케줄러와 발송 서비스는 정상 동작 중이며, 본 변경은 **프론트 권한/채널 구성** 범위로 한정된다.
- 캘린더 갱신 전략은 **탭 포커스 시 재조회**를 1차 해법으로 삼고, 전역 상태 공유를 통한 실시간 동기화는 본 스펙의 범위를 벗어난다(향후 확장 여지).
- 온보딩 완료 상태를 설정하는 명시적 액션은 클라이언트(온보딩 플로우 마지막 단계)에서 호출한다고 가정한다. 관리자/자동 완료 트리거는 범위 밖.
- 현재 사용자 프로필 API와 인증 체계는 재사용하며, 권한·인증 정책 변경은 없다.
